# Phase 12: App Shell Login And Owner Dashboard - Context

**รวบรวมเมื่อ:** 2026-06-03  
**สถานะ:** พร้อมสำหรับการวางแผนและการพัฒนา (Ready for Planning)

<domain>
## ขอบเขตของเฟส (Phase Boundary)

Phase 12 จะมุ่งเน้นการปรับปรุงหน้า Login, โครงสร้างการนำทาง (App Shell Navigation) บนจอคอมพิวเตอร์และมือถือ, และการแสดงผลข้อมูลบนแดชบอร์ด (Owner Dashboard) เพื่อสร้างความมั่นใจและความสะดวกในการใช้งานสูงสุดสำหรับผู้ดูแลระบบ โดยยังคงรักษาพฤติกรรมการทำงานของระบบเฟิร์มแวร์ปั๊ม, ระบบทำความเย็น (Cooling), Wi-Fi, และ SoftAP ที่มีความเสถียรในเวอร์ชัน v1.1 ไว้ทุกประการ (ไม่มีการแก้ไขตรรกะระดับล่างของคอนโทรลเลอร์)

ผลลัพธ์ของเฟสนี้คือการสร้างและปรับแต่ง Primitive ทางด้าน Accessibility (การจัดแต่ง `:focus-visible` และการนำส่งข้อมูลผ่าน `aria-live` หรือ `role="alert"`) ซึ่งจะถูกนำไปใช้ต่อในหน้าต่าง ๆ ของ Phase 13 ต่อไป

</domain>

<decisions>
## การตัดสินใจในการออกแบบและพัฒนา (Implementation Decisions)

### 1. การจัดการโฟกัสและ Accessibility (Focus Ring & Accessible Semantics)
- **D-01 (CSS :focus-visible):** ใช้คุณสมบัติ `:focus-visible` ใน `style.css` สำหรับตกแต่งปุ่ม (buttons), ลิงก์ (links), อินพุตฟิลด์ (inputs), ซีเล็คบ็อกซ์ (selects), เช็คบ็อกซ์ (checkboxes), และรายการเมนูทุกจุด ให้มีวงแหวนขอบสีฟ้าขนาด 3px (`--primary` หรือสีฟ้าสดใสชัดเจน) โดยจะแสดงเฉพาะเมื่อผู้ใช้กดใช้งานผ่านคีย์บอร์ด (Tabbing) เพื่อไม่ให้หน้าเว็บดูไม่สะอาดสำหรับผู้ใช้เมาส์ทั่วไป
- **D-02 (Live Regions & Alert):** ปรับปรุงโครงสร้าง HTML ในหน้า `login.html` และ `dashboard.html` รวมถึง JS ใน `app.js` ให้แสดงและประกาศเหตุการณ์ต่าง ๆ แก่เครื่องอ่านหน้าจอ (Screen Reader) โดยใช้ `aria-live="polite"` สำหรับข้อมูลทั่วไปหรือข้อมูลที่กำลังโหลด/ซิงก์สำเร็จ และใช้ `role="alert"` สำหรับข้อผิดพลาดฉุกเฉินหรือรหัสผ่านผิด

### 2. โครงสร้างการนำทางบนมือถือ (Mobile Navigation Pattern)
- **D-03 (Slide-out Drawer):** ปรับปรุงส่วนนำทางบนมือถือ (.mobile-topbar และสร้างแผง Drawer เพิ่มเติม) โดยใช้รูปแบบ Hamburger Menu (`&#9776;` หรือไอคอน 3 ขีด) ที่ด้านบน ซึ่งเมื่อกดแล้วจะเปิดแผงสไลด์ด้านข้าง (Slide-out Navigation Drawer) จากฝั่งซ้ายของจอ แผงเมนูนี้จะมีหน้าตาและลิงก์เชื่อมต่อไปยังทุก ๆ หน้าเหมือนกับ Sidebar บน Desktop (แดชบอร์ด, สถานะระบบ, การตั้งค่า Wi-Fi, เมนูติดตั้งฮาร์ดแวร์, และปุ่มออกจากระบบ)
- **D-04 (Consistent pattern):** อัปเดตโครงสร้าง Navigation Pattern นี้ให้เหมือนกันในทุก ๆ หน้าที่ต้องล็อกอิน (`dashboard.html`, `hardware.html`, `wifi.html`, `status.html`) เพื่อให้ผู้ใช้เข้าถึงทุกเมนูได้เท่ากันไม่ว่าจะอยู่ที่หน้าใด

### 3. การจัดลำดับความสำคัญของแดชบอร์ด (Dashboard Visual Hierarchy)
- **D-05 (Pump Runtime as Hero Section):** ปรับแต่งการแสดงผลในหน้าแดชบอร์ด โดยให้ส่วนของ "Pump Runtime" (สถานะปั๊ม, ตัวเลขเลขนับถอยหลัง) มีขนาดตัวอักษรใหญ่และเด่นชัดที่สุดด้วยดีไซน์ Hero-like เพื่อให้ผู้ใช้ตรวจสอบความปลอดภัยและตรรกะของปั๊มได้ในพริบตา
- **D-06 (Separate Cooling Channel):** แยกแผงควบคุมและแสดงผล "Cooling Runtime" ออกมาให้เป็นช่องสถานะระดับรองที่ชัดเจน ไม่ปะปนกับปั๊ม และตกแต่งสไตล์ให้เบากว่าส่วนของปั๊ม
- **D-07 (Secondary Settings & Diagnostics):** ปรับปรุงหน้าตาของฟอร์มกรอกค่าติดตั้งปั๊มและ Cooling รวมถึงข้อมูลสถิติเบื้องต้น (Quick Status) ให้มีน้ำหนักทางสายตารองลงมา และแยกแยะกลุ่มปุ่มใช้งานประจำวัน (เช่น Start, Stop, Sync) ออกจากแบบฟอร์มการตั้งค่าอย่างชัดเจน
- **D-08 (Removal of Accent Stripes):** ยกเลิกการใช้แถบหนาด้านข้างแผง (เช่น `border-left: 4px`) ใน Sidebar และกล่องสถานะ เพื่อลดความหนาแน่นทางสายตาตามเกณฑ์ของ Impeccable Audit

### 4. ภาษาของสถานะและข้อจำกัดแบบออฟไลน์ (Offline-First Footprint)
- **D-09 (Canonical Language):** ยึดรูปแบบภาษาไทยกึ่งทางการตามเกณฑ์ `11-STATE-LANGUAGE.md` อย่างเคร่งครัด
- **D-10 (No External Assets):** ห้ามเรียกใช้ CDN, Web Fonts (เช่น Google Fonts), หรือเครื่องมือตกแต่งภายนอกทุกรูปแบบ ทรัพยากรทั้งหมดต้องฝังอยู่ในอุปกรณ์ ESP32 และเรียกใช้งานได้แม้อุปกรณ์จะทำงานในโหมด SoftAP ที่ไม่มีอินเทอร์เน็ต

</decisions>

<code_context>
## การตรวจสอบจุดเชื่อมโยงในโค้ด (Existing Code Insights)

- โค้ด JavaScript ใน `main/static/app.js` ปัจจุบันมีตัวแปรสถานะและฟังก์ชันสำหรับอัปเดต UI เช่น `initLogin`, `initDashboard`, `syncPumpStatus`, และ `syncCoolingStatus` ที่จะนำมาผูกกับ `aria-live` และตรรกะเปิด/ปิด Hamburger Drawer
- โค้ด CSS ใน `main/static/style.css` มีตัวแปรแบรนด์ของระบบอยู่แล้ว เช่น `--primary` (`#0058bc`), `--bg-body` (`#F2F2F7`) และสไตล์สำหรับปุ่ม/ทาสก์บาร์ ซึ่งเราจะเข้าไปจัดระเบียบและลดความซ้ำซ้อนเพื่อคุมขนาดไฟล์ไม่ให้บวมขึ้นจนเบียดบังพื้นที่เก็บข้อมูลใน ESP32

</code_context>

<specifics>
## รายละเอียดเชิงลึกเฉพาะงาน (Specific Ideas)

- ปุ่มเปิดปิด Hamburger Menu บนมือถือจะใช้ตัวอักษร Unicode `&#9776;` (TRIGRAM FOR HEAVEN) เพื่อความเบา โดยไม่ต้องใช้ไฟล์ไอคอนภายนอก
- แผงนำทางบนมือถือ (Slide-out Drawer) จะใช้ CSS Class ในการควบคุมความกว้างและการสไลด์เข้าออก เช่น `.mobile-drawer.open` โดยจะมีการใช้ JavaScript สั้น ๆ ใน `app.js` ช่วยจัดการอีเวนต์เปิด/ปิด

</specifics>

## รายการที่เลื่อนออกไป (Deferred Ideas)

- การจัดสไตล์หน้าลงทะเบียนอุปกรณ์ (Install/Hardware), หน้าตั้งค่า Wi-Fi และหน้าสถานะแบบละเอียดจะคงโครงสร้างตรรกะการเชื่อมต่อเดิมไว้ และจะไปแต่งสไตล์โดยสมบูรณ์ใน Phase 13 ถัดไป

</deferred>
