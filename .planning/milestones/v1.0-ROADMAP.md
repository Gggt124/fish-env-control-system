## Overview

First shipped version of the Fish Pump Relay Timer Control firmware. A complete ESP32 pump controller with binary float switch input, relay output, two configurable timers, NVS persistence, authenticated web API, local dashboard UI, and hardware validation on real ESP32 DevKit V1 hardware.

## Phases

### Phase 1: Hardware-Safe Pump Control Core

**Goal:** Add the pump-control component with safe GPIO defaults, relay inactive initialization, timer state machine, and build validation.
**Depends on:** Existing Wi-Fi/web foundation
**Plans:** 3 plans

Plans:

- [x] 01-01: Pump hardware and timer defaults in `app_config.h`
- [x] 01-02: Reusable `pump_control` component with GPIO, debounce, timer state machine
- [x] 01-03: Boot integration and ESP-IDF build validation

**Details:**
Established GPIO32 as active-low float input with internal pull-up and GPIO26 as active-low relay output. Timer 1 defaults to ON 20s / OFF 1:00, Timer 2 to ON 10s / OFF 3:00. Relay initializes inactive. Float OFF selects Timer 1, Float ON selects Timer 2. Timer alternates ON/OFF phases. Stop forces relay inactive immediately.

### Phase 2: Pump Settings Persistence And Boot Behavior

**Goal:** Persist timer settings, relay polarity, and auto-start behavior in NVS, with safe defaults.
**Depends on:** Phase 1
**Plans:** 3 plans

Boot loads settings before pump init and conditionally auto-starts only when settings are valid. Auto-start defaults to disabled (changed from initial enabled default after quick task 260520-s4a for safety).

### Phase 3: Authenticated Pump Control API

**Goal:** Expose authenticated JSON APIs for pump config, pump status, start, and stop.

## Milestone Summary

**Key Decisions:**

- Decision: Use GPIO32 for float input with internal pull-up to GND (Rationale: Conservative pin avoiding boot strapping pins)
- Decision: Use GPIO26 for relay output (Rationale: General-purpose output avoiding common programming pins)
- Decision: Make relay polarity configurable (Rationale: Relay modules vary in active-high/low wiring)
- Decision: Auto-start disabled by default after hardware validation (Rationale: Safer boot behavior; user can enable via UI)
- Decision: Keep pump APIs under `/api/pump/*` (Rationale: Separate pump contracts from system/Wi-Fi routes)
- Decision: Dashboard as pump control surface (Rationale: Operator needs timer setup and runtime state first)

**Issues Resolved:**

- ESP-IDF 6.1 compatibility: `esp_driver_gpio` component dependency instead of legacy `driver`
- Build timeout/stale ninja process during long ESP-IDF builds
- Source/planning drift on auto-start default resolved at milestone close

**Issues Deferred:**

- Automated hardware-in-loop testing (no test rig exists)
- Automated visual/browser verification (Playwright unavailable)
- Nyquist validation artifacts (`*-VALIDATION.md`) for all phases

**Technical Debt Incurred:**

- Manual hardware evidence as primary validation source for Phase 5
- App partition nearly full (~5% free) after OTA partition layout change
- No automated screenshot/UI testing for Phase 4
