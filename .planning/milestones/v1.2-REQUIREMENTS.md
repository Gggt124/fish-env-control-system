## v1.2 Requirements — All Complete

### Owner Dashboard

- [x] **UI-08**: Owner can identify the main pump state, selected timer and relay, current phase, countdown, and float switch state clearly from the dashboard. — **Complete** (Phase 12)
- [x] **UI-09**: Owner can identify temperature, cooling mode, cooling relay state, and sensor fault state as a separate operational channel from the pump relays. — **Complete** (Phase 12)
- [x] **UI-10**: Owner can distinguish daily operation controls from configuration settings and secondary diagnostics without scanning equal-weight panels. — **Complete** (Phase 12)

### App Shell And Login

- [x] **UI-14**: User can understand login purpose, credential entry, loading, and authentication error recovery clearly on desktop and narrow mobile screens. — **Complete** (Phase 12)
- [x] **UI-15**: Authenticated user can navigate dashboard, Hardware/Install, Wi-Fi, Status, and logout consistently from the app shell on desktop and narrow mobile screens. — **Complete** (Phase 12)

### Hardware Install

- [x] **HW-06**: Installer can see explicit guidance to add an external `4.7 kOhm` pull-up resistor between DS18B20 DQ and `3.3 V` before the next hardware cycle. — **Complete** (Phase 13)
- [x] **UI-11**: Installer can distinguish active GPIO values, pending GPIO values, and reboot-required state clearly on the Hardware/Install page. — **Complete** (Phase 13)
- [x] **UI-12**: Installer can use the Hardware/Install flow on a narrow mobile screen while retaining safe dropdown options and the reboot/wiring confirmation checkbox. — **Complete** (Phase 13)

### Setup States

- [x] **SETUP-01**: User can identify loading, error, empty, disconnected, and connected states during Wi-Fi scan and connect flows. — **Complete** (Phase 13)
- [x] **SETUP-02**: User can identify loading, success, error, disabled, and pending-reboot states on dashboard and Hardware/Install operations. — **Complete** (Phase 13)

### System Status

- [x] **UI-16**: User can scan system, memory, uptime, Wi-Fi station, access point, and service diagnostics clearly on desktop and narrow mobile screens. — **Complete** (Phase 13)

### Accessibility

- [x] **A11Y-01**: User can use login, app shell, dashboard, Hardware/Install, Wi-Fi, and Status pages on a narrow mobile viewport without losing access to primary content or actions. — **Complete** (Phase 14)
- [x] **A11Y-02**: Keyboard user can navigate interactive controls with visible focus treatment and understandable labels or instructions. — **Complete** (Phase 11)
- [x] **A11Y-03**: Assistive-technology user can recognize dynamic status and error messages programmatically. — **Complete** (Phase 11)

### Design System

- [x] **UI-13**: User experiences a consistent embedded product UI shaped by the existing `DESIGN.md` system and reviewed with `frontend-ui-ux`, `ui-ux-pro-max`, and `impeccable` guidance adapted to offline ESP32 constraints. — **Complete** (Phase 11)
