<objective>
Fix the blocker gap (UAT test 3) where the TFT background update task is created but never executes its first statement. The task is created with only 3072 bytes of stack on tft_display.c:475, then immediately deep-calls into `tft_display_draw_dashboard_skeleton` → `tft_clear` → `tft_fill_rect` which uses a 640-byte local buffer; combined with FreeRTOS/ESP_LOGI overhead the 3KB stack overflows before the first `ESP_LOGI` on line 292 can run. We will: (1) move the 640-byte `chunk_buf` out of task stack, (2) bump `xTaskCreate` stack from 3072 to 8192 bytes, (3) capture and log the `xTaskCreate` return value so any future creation failure is visible, and (4) enable `CONFIG_FREERTOS_CHECK_STACKOVERFLOW=2` so future stack overflows panic visibly instead of silently corrupting memory. After this plan, the splash → dashboard transition works (even if the dashboard is still visually corrupted — that is fixed in 18-04).
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| FreeRTOS Task Scheduler | The tft_display_task runs in its own thread. Stack overflow in this task could silently corrupt other tasks' stacks (including the main task running Wi-Fi/web/pump/cooling). |
| Heap | tft_fill_rect's 640-byte chunk_buf is currently on the task stack. Heavy use in deep call chains risks overflow. |
| Build system | The new CONFIG_FREERTOS_CHECK_STACKOVERFLOW=2 setting is a global safety net; a panic-on-overflow event would crash the device, which is acceptable for development but must be removed before any unattended deployment. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-18-05 | Denial of Service | main/tft_display.c | mitigate | (1) Move chunk_buf[320] to static so the 640 bytes are not on the task stack. (2) Bump xTaskCreate stack from 3072 to 8192 bytes. (3) Capture xTaskCreate return value and ESP_LOGE on failure. (4) Enable CONFIG_FREERTOS_CHECK_STACKOVERFLOW=2 so any future stack overflow panics visibly instead of corrupting memory. |
| T-18-05b | Information Disclosure | main/tft_display.c | mitigate | The chunk_buf is now file-scope static. Since the only writer is the TFT draw path and the draw path is serialized via s_trans_done_sem, there is no information leak between tasks. If a future contributor adds a concurrent writer they must add a mutex — leave a comment in the source. |
| T-18-05c | Denial of Service | sdkconfig.defaults | accept (dev only) | CONFIG_FREERTOS_CHECK_STACKOVERFLOW=2 causes a Guru Meditation panic on overflow. This is desirable in development to surface bugs, but must be reduced to 1 or 0 before unattended production deployment. Note this in DEVELOPMENT.md or a comment. |
| T-18-05-SC | Tampering | sdkconfig.defaults | mitigate | No new managed component dependencies are introduced by this plan. The only build-system change is a single integer Kconfig value. slopcheck N/A. |
</threat_model>
