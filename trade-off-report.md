# ระบบจัดการ Firmware: รายงานการแก้ไขปัญหาและการวิเคราะห์ Trade-off

รายงานฉบับนี้จัดทำขึ้นเพื่อบันทึกการตัดสินใจในการแก้ไขปัญหา (Fixes) ต่างๆ ในโค้ดของระบบ ควบคู่ไปกับการวิเคราะห์ผลกระทบที่จะเกิดขึ้น (Trade-offs) ทั้งสำหรับส่วนที่เลือกแก้ไขและส่วนที่ตัดสินใจละเว้นไว้ชั่วคราว

---

## 1. สิ่งที่ได้รับการแก้ไข

### 1.1 การเข้าคิวล็อก (Mutex/Semaphore) ของ `pump_control.c`
**ปัญหาที่พบ:**
การเรียกใช้ `s_pump_mutex` ในฟังก์ชันต่างๆ ใช้ `portMAX_DELAY` ซึ่งหมายถึง Task จะรอคอยแบบไม่มีที่สิ้นสุด หากเกิดกรณีที่ฮาร์ดแวร์ขัดข้องหรือ Task หนึ่ง Block นาน อาจทำให้ส่วนอื่นที่มารอ Mutex ตัวนี้ (เช่น Web Server ที่มาเรียกเช็ค Status) เกิดค้างตามไปด้วย

**การแก้ไข:**
เปลี่ยนการเรียก `xSemaphoreTake(s_pump_mutex, portMAX_DELAY)` เป็น `xSemaphoreTake(s_pump_mutex, pdMS_TO_TICKS(1000))` (ตั้งเวลารอสูงสุด 1 วินาที) หากไม่สามารถจับ Lock ได้ในเวลาที่กำหนด ระบบจะทำการ `return false` ออกไป เพื่อให้ Web API สามารถจัดการคืนค่า Error กลับไปหาผู้เรียก (เช่น HTTP 500) โดยที่ระบบไม่เกิดการ Hang/Deadlock

**Trade-off ที่เกิดขึ้น:**
- **ข้อดี:** ป้องกันการเกิด Deadlock หรือระบบค้างโดยสมบูรณ์ ทำให้ระบบมีความเสถียรมากขึ้น
- **ข้อควรระวัง:** อาจเกิด Error กลับไปยังผู้ใช้งานเป็นครั้งคราวในช่วงที่ระบบทำงานหนัก แต่ดีกว่าการปล่อยให้ระบบค้างถาวร

### 1.2 การจัดการ Strings ควบคุมขนาด (Buffer Overflows) ด้วย `strncpy`
**ปัญหาที่พบ:**
ในฟังก์ชันการตั้งค่า Wi-Fi และ IP Address (`web_server.c`) โค้ดเดิมใช้ `strncpy` ในการจำกัดความยาวของ String ซึ่งปลอดภัยจาก Buffer Overflow แต่วิธีนี้ใช้วิธีการ "ตัดทอน" (Truncate) String ที่ยาวเกินไปทิ้งแบบเงียบๆ ทำให้ระบบอาจทำงานผิดพลาดได้ (เช่น รหัสผ่านถูกตัดให้สั้นลง ทำให้ต่อ Wi-Fi ไม่ได้) แต่ผู้ใช้ไม่รู้สาเหตุ

**การแก้ไข:**
เพิ่มการตรวจสอบความยาว (`strlen`) ก่อนเรียกใช้ `strncpy` ในตัวแปรที่สำคัญ ได้แก่ `ssid`, `password`, `ip`, `gateway`, `netmask`, `dns` หากความยาวเกินกว่าที่ขนาดของ Buffer รองรับได้ (เช่น SSID ยาวกว่า 32 ตัวอักษร) จะไม่มีการบันทึก และจะคืนค่า `400 Bad Request` พร้อมข้อความชี้แจงสาเหตุกลับไปยังผู้ใช้งาน

**Trade-off ที่เกิดขึ้น:**
- **ข้อดี:** ผู้ใช้รู้ทันทีเมื่อกรอกข้อมูลผิดพลาดหรือยาวเกินไป ทำให้ปัญหาการตั้งค่าเพี้ยนเนื่องจาก Truncation หมดไป
- **ข้อควรระวัง:** การปรับปรุงนี้ทำขึ้นเฉพาะส่วน Backend ผู้พัฒนาระบบฝั่ง Frontend อาจต้องอัปเดตหน้าเว็บเพื่อแสดงผล Error Message จาก API ให้ผู้ใช้งานเห็นชัดเจนขึ้น

## 2. สิ่งที่ตัดสินใจละเว้น (ไม่ได้แก้ไขในรอบนี้)

### 2.1 การทำ HTTP API Authentication ไม่มี Rate Limiting ระดับ Global
**เหตุผลที่ไม่ทำ:**
การพัฒนาระบบ Global Rate Limiting / Backoff จะช่วยป้องกันปัญหาการโจมตีแบบ Brute-force ได้ แลกกับความซับซ้อนของการจัดการ Session / Token ที่สูงขึ้นและกินทรัพยากร CPU / Memory มากขึ้น ซึ่งอาจไม่จำเป็นหรือคุ้มค่าสำหรับอุปกรณ์ IoT ขนาดเล็กที่ไม่ได้เปิดให้ต่อกับเครือข่ายอินเทอร์เน็ตสาธารณะ (Public Internet)

### 2.2 ขนาดของ Task Stack (Stack Size Optimization)
**เหตุผลที่ไม่ทำ:**
แม้ว่าการลดขนาดของ Task Stack (`dns_server`, `tft_display_task` ฯลฯ) ที่ตั้งไว้ 4096 Bytes จะช่วยเพิ่มพื้นที่ RAM ว่าง (Free Heap) ได้ แต่มีความเสี่ยงสูงที่จะเกิดปัญหาร้ายแรงอย่าง Stack Overflow ที่จะส่งผลให้บอร์ด Reset หรือ Panic ได้ หากไม่มีการวัดค่า "High Water Mark" ในสภาพแวดล้อมและโหลดการทำงานจริงก่อนการปรับลด
